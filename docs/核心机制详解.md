## 1. 多候选生成与排名机制

### 1.1 什么是多候选（Multi-Candidates）？

**多候选**是指在**单次迭代**中，AutoChip 同时让 LLM 生成多个不同版本的 Verilog 代码，然后通过编译和仿真自动选择最好的那个。

**配置参数：** `"num_candidates": N`

- `N=1`：每次迭代生成 1 个版本（默认）
- `N=3`：每次迭代生成 3 个版本
- `N=5`：每次迭代生成 5 个版本（论文中的实验设置）

---

#### **步骤 4：calculate_rank() 的详细评估流程**

在 [`languagemodels.py`](file:///home/gq/Autochip_workspace/AutoChip/autochip_scripts/languagemodels.py) 的 Line 443-497：

```python
def calculate_rank(self, outdir, module, testbench):
    filename = os.path.join(outdir, module+".sv")
    vvp_file = os.path.join(outdir, module+".vvp")

    # 🔧 第 1 步：编译检查
    compiler_cmd = f"iverilog -Wall -Winfloop -Wno-timescale -g2012 -s tb -o {vvp_file} {filename} {testbench}"
    comp_return, comp_err, comp_out = vh.compile_iverilog(outdir, module, compiler_cmd, self)

    if comp_return != 0:
        # ❌ 编译失败
        self.rank = -1
        self.message = "The design failed to compile..."
        return

    elif comp_err != "":
        # ⚠️ 编译警告
        self.rank = -0.5
        self.message = "The design compiled with warnings..."

    else:
        # ✅ 编译成功，进入仿真阶段
        sim_return, sim_err, sim_out = vh.simulate_iverilog(simulator_cmd)
        
        # 解析仿真输出，提取 Mismatches 和 Samples
        mismatch_pattern = r"Mismatches: (\d+) in (\d+) samples"
        match = re.search(mismatch_pattern, sim_out.splitlines()[-1])
        
        mismatches = int(match.group(1))
        samples = int(match.group(2))
        
        if mismatches > 0:
            # ⚠️ 仿真有部分不匹配
            self.rank = (samples - mismatches) / samples
            # 例如：mismatches=5, samples=100 → rank = 0.95
        else:
            # ✅ 完美通过
            self.rank = 1.0
```

**Rank 评分规则总结：**

| Rank 值 | 条件 | 含义 |
|---------|------|------|
| **-2** | 无法从 LLM 回复中提取 Verilog 代码 | 格式错误 |
| **-1** | Icarus Verilog 编译失败 | 语法错误 |
| **-0.5** | 编译成功但有警告 | 潜在问题 |
| **0.0 ~ 0.99** | 编译成功，仿真有不匹配 | 功能部分正确 |
| **1.0** | 编译成功 + 仿真 100% 匹配 | 完美通过 |

#### **步骤 5：排名并选择最佳候选**

Line 286-291：

```python
## RANK RESPONSES
# 🔑 关键排序逻辑：先按 rank 降序，再按代码长度升序
max_rank_response = max(responses, key=lambda resp: (resp.rank, -resp.parsed_length))

# 更新全局最优响应
if max_rank_response.rank > global_max_response.rank:
    global_max_response = max_rank_response
elif max_rank_response.rank == global_max_response.rank and max_rank_response.parsed_length > global_max_response.parsed_length:
    global_max_response = max_rank_response

print(f"Response ranks: {[resp.rank for resp in responses]}")
print(f"Response lengths: {[resp.parsed_length for resp in responses]}")
```

**排序规则详解：**

```python
key=lambda resp: (resp.rank, -resp.parsed_length)
```

这是一个**元组排序**，优先级从高到低：

1. **第一优先级：`resp.rank`（越高越好）**
   - 1.0 > 0.95 > -0.5 > -1 > -2
   
2. **第二优先级：`-resp.parsed_length`（负号表示越短越好）**
   - 当两个候选的 rank 相同时，选择**代码更短**的那个
   - 原因：更短的代码通常意味着更简洁、更高效的设计

**示例：**

假设有 3 个候选：
```
候选 0: rank = 1.0, length = 350
候选 1: rank = 1.0, length = 420
候选 2: rank = -1,  length = 280
```

排序结果：
1. **候选 0**（rank=1.0, length=350）✅ **被选中**
2. 候选 1（rank=1.0, length=420）
3. 候选 2（rank=-1, length=280）

虽然候选 2 最短，但它的 rank 太低（编译失败），所以不会被选中。

### 1.3 多候选的目录结构

假设 `num_candidates=3`，迭代 2 次，目录结构如下：

```
output_dir/
├── iter0/                    # 第 1 次迭代
│   ├── response0/            # 候选 0
│   │   ├── top_module.sv
│   │   ├── top_module.vvp
│   │   └── log.txt           # 包含 rank=-1（编译失败）
│   ├── response1/            # 候选 1
│   │   ├── top_module.sv
│   │   ├── top_module.vvp
│   │   └── log.txt           # 包含 rank=0.95（部分匹配）
│   └── response2/            # 候选 2 ✅ 被选中
│       ├── top_module.sv
│       ├── top_module.vvp
│       └── log.txt           # 包含 rank=1.0（完美）
└── iter1/                    # 第 2 次迭代（基于 iter0/response2 的反馈）
    ├── response0/
    ├── response1/
    └── response2/
```

**关键点：**
- 每个候选都有**独立的输出目录**
- 每个候选都会**独立编译和仿真**
- 只有**最佳的候选**会进入下一轮迭代

### 1.4 多候选的优势与代价

#### **优势：**
✅ **提高成功率：** LLM 具有随机性，多次生成可能得到更好的结果  
✅ **多样性探索：** 不同的实现思路可能有不同的优劣  
✅ **自动化选择：** 无需人工干预，系统自动选择最佳方案  

#### **代价：**
❌ **API 成本增加：** `num_candidates=3` 意味着每次迭代调用 3 次 API  
❌ **时间开销增加：** 需要编译和仿真 3 个版本  
❌ **存储空间增加：** 每个候选都保存独立的文件  

**建议：**
- 简单任务：`num_candidates=1`（节省成本）
- 复杂任务：`num_candidates=3~5`（提高成功率）
- 研究实验：`num_candidates=10+`（探索多样性）

## 2. 混合模型策略详解

### 2.1 什么是混合模型（Mixed Models）？

**混合模型策略**是指在**不同的迭代阶段**使用**不同的 LLM 模型**，以平衡成本和性能。

**典型策略：**
- **前期迭代（0-2）：** 使用便宜但较弱的模型（如 GPT-4o-mini）
- **后期迭代（3+）：** 切换到昂贵但强大的模型（如 GPT-4o）

**目的：**
- 💰 **降低成本：** 大部分简单修复用便宜模型完成
- 🚀 **提高成功率：** 疑难问题交给强模型处理
- 📊 **优化性价比：** 论文中提到可降低 60% 的 Token 支出

---

### 2.2 混合模型的配置方式

**只能在 JSON 配置文件中设置**，不支持命令行参数。

**配置文件示例：**

```json
{
    "general": {
        "prompt": "../verilogeval_prompts_tbs/validation_set/rule90/rule90.sv",
        "name": "top_module",
        "testbench": "../verilogeval_prompts_tbs/validation_set/rule90/rule90_tb.sv",
        "iterations": 5,
        "num_candidates": 1,
        "mixed-models": true,
        "outdir": "output_mixed",
        "log": "log.txt"
    },
    "mixed-models": {
        "model1": {
            "start_iteration": 0,
            "model_family": "ChatGPT",
            "model_id": "gpt-4o-mini"
        },
        "model2": {
            "start_iteration": 3,
            "model_family": "ChatGPT",
            "model_id": "gpt-4o"
        }
    }
}
```

**配置说明：**

| 字段 | 含义 | 示例值 |
|------|------|--------|
| `start_iteration` | 从第几次迭代开始使用该模型 | 0, 3, -1 |
| `model_family` | 模型家族 | "ChatGPT", "Claude", "Siliconflow" |
| `model_id` | 具体模型 ID | "gpt-4o-mini", "gpt-4o" |

**特殊值：**
- `start_iteration: 0` → 从第一次迭代就开始使用
- `start_iteration: -1` → 最后一次迭代使用（动态计算）
- `start_iteration: 3` → 从第 4 次迭代（iter3）开始使用

### 2.3 混合模型的工作流程

#### **步骤 1：配置验证**

在 [`config_handler.py`](file:///home/gq/Autochip_workspace/AutoChip/autochip_scripts/config_handler.py) 的 `validate_mixed_model_config()` 函数中：

```python
def validate_mixed_model_config(mixed_model_config, max_iterations):
    seen_start_iterations = set()
    adjusted_config = {}
    has_start_at_zero = False

    for model_name, model_info in mixed_model_config.items():
        start_iteration = model_info['start_iteration']

        # 调整负数索引
        if start_iteration < 0:
            start_iteration += max_iterations + 1
            # 例如：max_iterations=5, start_iteration=-1 → 5

        # 检查范围
        if not (0 <= start_iteration <= max_iterations):
            raise ValueError(f"Invalid start_iteration...")

        # 检查冲突
        if start_iteration in seen_start_iterations:
            raise ValueError(f"Conflicting start_iteration...")
        
        seen_start_iterations.add(start_iteration)

        # 必须有模型从 iteration 0 开始
        if start_iteration == 0:
            has_start_at_zero = True

    if not has_start_at_zero:
        raise ValueError("No model starting at iteration 0...")

    return adjusted_config
```

**验证规则：**
1. ✅ 必须有一个模型从 `iteration 0` 开始
2. ✅ 不能有多个模型使用相同的 `start_iteration`
3. ✅ `start_iteration` 必须在有效范围内（0 到 max_iterations）

---

#### **步骤 2：根据当前迭代选择模型**

在 [`verilog_handling.py`](file:///home/gq/Autochip_workspace/AutoChip/autochip_scripts/verilog_handling.py) 的 `get_iteration_model()` 函数中（Line 173-182）：

```python
def get_iteration_model(iteration, mixed_model_config):
    # 按 start_iteration 降序排序
    sorted_models = sorted(mixed_model_config.values(), 
                          key=lambda x: x['start_iteration'], 
                          reverse=True)

    family = None
    model_id = None
    
    # 找到第一个 start_iteration <= 当前 iteration 的模型
    for model_info in sorted_models:
        if iteration >= model_info['start_iteration']:
            family = model_info['model_family']
            model_id = model_info['model_id']
            break  # 找到后立即返回

    return family, model_id
```

**工作原理：**

假设配置了 3 个模型：
```json
{
    "model1": {"start_iteration": 0, "model_id": "gpt-4o-mini"},
    "model2": {"start_iteration": 2, "model_id": "gpt-4"},
    "model3": {"start_iteration": 4, "model_id": "claude-3-opus"}
}
```

排序后（降序）：
```
model3: start_iteration=4
model2: start_iteration=2
model1: start_iteration=0
```

**选择逻辑：**

| 当前 Iteration | 选择的模型 | 原因 |
|---------------|-----------|------|
| 0 | gpt-4o-mini | 0 >= 0（model1） |
| 1 | gpt-4o-mini | 1 >= 0（model1），但 1 < 2（model2） |
| 2 | gpt-4 | 2 >= 2（model2） |
| 3 | gpt-4 | 3 >= 2（model2），但 3 < 4（model3） |
| 4 | claude-3-opus | 4 >= 4（model3） |
| 5 | claude-3-opus | 5 >= 4（model3） |

**核心思想：** 选择**最近的已启动模型**。

### 2.4 混合模型的实际案例

#### **案例 1：两阶段策略**

```json
{
    "mixed-models": {
        "cheap_model": {
            "start_iteration": 0,
            "model_family": "ChatGPT",
            "model_id": "gpt-4o-mini"
        },
        "strong_model": {
            "start_iteration": 3,
            "model_family": "ChatGPT",
            "model_id": "gpt-4o"
        }
    }
}
```

**执行流程：**
```
Iter 0: gpt-4o-mini  → 生成代码 → 编译失败（Rank -1）
Iter 1: gpt-4o-mini  → 修复代码 → 仍有错误（Rank -1）
Iter 2: gpt-4o-mini  → 再次修复 → 编译警告（Rank -0.5）
Iter 3: gpt-4o       → 强力修复 → 完美通过（Rank 1.0）✅
```

**成本分析：**
- 前 3 次迭代使用便宜模型（$0.002/次）
- 最后 1 次迭代使用强模型（$0.01/次）
- 总成本：3×$0.002 + 1×$0.01 = $0.016
- 如果全程使用 gpt-4o：4×$0.01 = $0.04
- **节省成本：60%**

### 2.5 混合模型的注意事项

#### **⚠️ 限制 1：只能通过配置文件使用**

命令行参数不支持混合模型：
```bash
# ❌ 错误：命令行无法指定混合模型
python generate_verilog.py -f ChatGPT -m gpt-4o-mini

# ✅ 正确：使用配置文件
python generate_verilog.py -c config_mixed.json
```

#### **⚠️ 限制 2：必须有模型从 iteration 0 开始**

```json
{
    "mixed-models": {
        "model1": {
            "start_iteration": 2,  // ❌ 错误：没有模型从 0 开始
            "model_id": "gpt-4o"
        }
    }
}
```

#### **⚠️ 限制 3：不能有两个模型使用相同的 start_iteration**

```json
{
    "mixed-models": {
        "model1": {"start_iteration": 2, "model_id": "gpt-4"},
        "model2": {"start_iteration": 2, "model_id": "claude"}  // ❌ 冲突
    }
}
```

#### **💡 最佳实践**

1. **简单任务：** 不需要混合模型，单一模型即可
2. **中等难度：** 两阶段策略（便宜 → 强）
3. **复杂任务：** 三阶段策略（弱 → 中 → 强）
4. **成本控制：** 尽量延长便宜模型的使用轮次
5. **成功率优先：** 在关键迭代提前切换到强模型

```
│     Iteration 0                          │
│                                         │
│  并行/串行生成 3 个候选：                 │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐│
│  │ 候选 0   │ │ 候选 1   │ │ 候选 2   ││
│  └────┬─────┘ └────┬─────┘ └────┬─────┘│
│       ↓            ↓            ↓       │
│  独立编译+仿真  独立编译+仿真  独立编译+仿真│
│       ↓            ↓            ↓       │
│  rank = -1    rank = 0.95  rank = 1.0  │
│  (编译失败)   (部分匹配)   (完美) ✅    │
└──────────────┬──────────────────────────┘
               ↓
┌─────────────────────────────────────────┐
│     排名并选择最佳候选                    │
│                                         │
│  max_rank_response = max(responses,     │
│      key=lambda r: (r.rank, -r.length)) │
│                                         │
│  结果：候选 2 被选中（rank=1.0）         │
└──────────────┬──────────────────────────┘
               ↓
┌─────────────────────────────────────────┐
│     只有候选 2 进入下一轮迭代             │
│  conv.add_message("assistant",          │
│      candidate_2.code)                  │
└─────────────────────────────────────────┘
```

## 4. 实际案例分析

### 4.1 案例：为什么 demo_output 只用了 iter0？

**回顾：** `demo_output` 目录中只有 `iter0`，且 Rank = 1.0。

**原因分析：**

1. **使用了 GPT-4o-mini 模型**
   - 代码生成能力强
   - 一次性生成了正确的语法

2. **num_candidates = 1**
   - 只生成了 1 个候选
   - 该候选直接通过了编译和仿真

3. **首次迭代就成功**
   ```python
   if max_rank_response.rank == 1:
       success = True  # ← 立即退出循环
   ```
   - `while not (success or timeout)` 条件不再满足
   - 不会进入 iter1

**日志证据：**
```
Time to Generate: 5.262348651885986
Best ranked response at iteration 0 with response number 0
Rank of best response: 1.0
```

### 4.2 案例：如果设置 num_candidates=3 会发生什么？

**假设场景：** 使用相同的配置，但 `num_candidates=3`

**可能的结果：**

```
Iteration 0:
  候选 0: rank = 1.0, length = 350  ✅ 被选中
  候选 1: rank = 1.0, length = 380
  候选 2: rank = -1,  length = 290  (编译失败)

Response ranks: [1.0, 1.0, -1]
Response lengths: [350, 380, 290]

Best ranked response at iteration 0 with response number 0
```

**分析：**
- 候选 0 和候选 1 都完美通过（rank=1.0）
- 系统选择**代码更短**的候选 0（350 < 380）
- 仍然只需要 iter0，因为已经成功

**成本对比：**
- `num_candidates=1`：1 次 API 调用
- `num_candidates=3`：3 次 API 调用
- **成本增加 3 倍**，但成功率可能提高

## 🎯 总结对比表

| 特性 | 单候选 | 多候选 | 混合模型 |
|------|--------|--------|----------|
| **定义** | 每次迭代生成 1 个版本 | 每次迭代生成 N 个版本 | 不同迭代使用不同模型 |
| **配置参数** | `num_candidates=1` | `num_candidates=N` | `mixed-models` JSON |
| **评估方式** | 编译+仿真 1 次 | 编译+仿真 N 次 | 根据迭代动态选择模型 |
| **选择标准** | 唯一候选自动选中 | `max(rank, -length)` | `start_iteration` 决定 |
| **优势** | 成本低，速度快 | 成功率高，多样性好 | 平衡成本与性能 |
| **劣势** | 成功率依赖单次生成 | 成本高，耗时长 | 配置复杂，仅支持 JSON |
| **适用场景** | 简单任务 | 复杂任务 | 长期项目，成本敏感 |

## 💡 最佳实践建议

### 1. 如何选择 num_candidates？

| 任务难度 | 推荐值 | 原因 |
|---------|--------|------|
| 简单（如 mux2to1） | 1 | 一次成功概率高 |
| 中等（如 rule90） | 2-3 | 平衡成本与成功率 |
| 复杂（如 FSM、CPU） | 5-10 | 需要多样性探索 |
| 研究实验 | 10+ | 统计分析需要 |

### 2. 如何设计混合模型策略？

**两阶段策略（推荐）：**
```json
{
    "mixed-models": {
        "cheap": {"start_iteration": 0, "model_id": "gpt-4o-mini"},
        "strong": {"start_iteration": 3, "model_id": "gpt-4o"}
    }
}
```

**三阶段策略（复杂任务）：**
```json
{
    "mixed-models": {
        "weak": {"start_iteration": 0, "model_id": "gpt-3.5-turbo"},
        "medium": {"start_iteration": 2, "model_id": "gpt-4o-mini"},
        "strong": {"start_iteration": 4, "model_id": "gpt-4o"}
    }
}
```

### 3. 排名规则的深层理解

```python
key=lambda resp: (resp.rank, -resp.parsed_length)
```

**为什么要在 rank 相同时选择更短的代码？**

1. **简洁性原则：** 更短的代码通常意味着更高效的设计
2. **减少错误概率：** 代码越少，潜在 bug 越少
3. **资源节约：** 短代码占用更少的硬件资源（LUT、FF 等）
4. **可维护性：** 简洁的代码更易理解和修改

**例外情况：**
- 如果短代码牺牲了可读性或鲁棒性，可能需要人工审查
- 某些设计（如流水线）本身就需要较长的代码
